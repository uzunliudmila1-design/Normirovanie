## О проекте

Веб-приложение для автоматического расчёта норм времени производственных операций (t_шт — штучное время, t_пз — подготовительное время) на машиностроительном заводе. Технолог загружает PDF маршрутной карты (обязательно) и чертёж детали (необязательно), Claude анализирует документы, подбирает оборудование из базы завода и предлагает нормы, которые технолог может скорректировать и сохранить в БД.

## Запуск

```bash
# Через скрипт (рекомендуется, настраивает UTF-8)
./run.sh

# Или напрямую
python3 app.py
```

Приложение запускается на порту **5050**. Debug-режим включён, автоперезагрузка отключена (`use_reloader=False`).

## Настройка окружения

Скопировать `.env.example` в `.env` и задать переменные:

```
ANTHROPIC_API_KEY=sk-ant-...
USE_STUB=false          # true — тестовые данные без API
USE_CLAUDE_CODE=true    # true — Claude Code CLI, false — Anthropic REST API
```

## Установка зависимостей

```bash
python3 -m venv venv
venv/bin/pip install -r requirements.txt
```

`run.sh` использует виртуальное окружение в `/tmp/norm_venv`, а не в `venv/`.

## Архитектура

Одностраничное приложение: Flask бэкенд + ванильный JS фронтенд.

```
templates/index.html  →  fetch /api/analyze  →  app.py  →  Anthropic API или Claude Code CLI
                          fetch /api/confirm  →  norming.db (SQLite)
                          fetch /api/history  →  norming.db
```

**app.py** — единственный бэкенд-файл (~535 строк):
- `init_db()` — создаёт таблицу `нормы` в SQLite при первом запуске
- `load_equipment_text()` — загружает `equipment_db.json`, фильтрует рабочее оборудование, форматирует текст для промпта, кешируется в `_EQUIPMENT_TEXT`
- `analyze_with_claude_code()` — сохраняет PDF во временные файлы, вызывает `claude -p` (путь к бинарнику захардкожен: `CLAUDE_BIN = /Users/liudmila/Library/Application Support/Claude/claude-code/2.1.78/...`)
- `analyze_with_claude()` — кодирует PDF в base64, отправляет через Anthropic SDK, модель `claude-sonnet-4-20250514`
- `/api/analyze` — принимает PDF маршрутной карты + опционально чертёж, выбирает метод анализа по переменным окружения, возвращает JSON список операций
- `/api/confirm` — сохраняет подтверждённые нормы в БД
- `/api/history` — возвращает последние 200 записей

**templates/index.html** — весь фронтенд (~1215 строк): drag-and-drop загрузка PDF, таблица результатов с редактируемыми полями (включая столбец «Оборудование»), история.

## База данных

SQLite файл `norming.db`. Схема создаётся автоматически при первом запуске через `init_db()`. Колонки `режимы`, `изделие`, `оборудование` добавляются через `ALTER TABLE` (миграция для существующих БД):

```sql
CREATE TABLE нормы (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    деталь TEXT NOT NULL,
    операция TEXT NOT NULL,
    оборудование TEXT,
    t_шт_предложено REAL,
    t_шт_подтверждено REAL,
    t_пз_предложено REAL,
    t_пз_подтверждено REAL,
    режимы TEXT,
    обоснование TEXT,
    изделие TEXT,
    дата TEXT,
    подтверждено INTEGER DEFAULT 0
)
```

## Логика работы программы

### 1. Запуск и инициализация

1. `app.py` стартует → `init_db()` создаёт `norming.db` + применяет миграции
2. `load_equipment_text()` загружает `equipment_db.json` (база оборудования завода) — вызывается лениво при первом запросе `/api/analyze`, результат кешируется в `_EQUIPMENT_TEXT`
3. Из `equipment_db.json` фильтруются только рабочие единицы (статусы: `Рабочий`, `Работает`, `По требованию`, `Резерв`, `Нет шильда`, `""`), затем группируются по типу, дедуплицируются по первым 40 символам названия → форматируется текстовый блок для системного промпта

### 2. Загрузка файлов (фронтенд → `/api/analyze`)

Пользователь в браузере:
- Загружает **маршрутную карту** (PDF, обязательно)
- Загружает **чертёж детали** (PDF, необязательно)
- Нажимает «Анализировать»

`fetch POST /api/analyze` отправляет `multipart/form-data` с полями: `marshrutnaya`, `chertezh` (опц.).

> Поля «Типовой техпроцесс» и «Размер партии» удалены из интерфейса. `batch_size` зафиксирован как 1 на фронтенде.

### 3. Выбор метода анализа (app.py)

Переменная окружения определяет путь:

```
USE_STUB=true        → возвращает STUB_OPERATIONS (4 тестовые операции), API не вызывается
USE_CLAUDE_CODE=true → analyze_with_claude_code()  — вызов Claude Code CLI
USE_CLAUDE_CODE=false→ analyze_with_claude()        — вызов Anthropic REST API
```

### 4a. Анализ через Claude Code CLI (`analyze_with_claude_code`)

1. PDF-файлы сохраняются во **временные файлы** (`tempfile.NamedTemporaryFile`, суффикс `.pdf`)
2. Формируется текстовый промпт с путями к временным файлам + размер партии + блок оборудования
3. Вызывается subprocess:
   ```
   claude -p "<промпт>" \
     --allowedTools Read \
     --output-format json \
     --system-prompt "<SYSTEM_PROMPT>" \
     --model sonnet \
     --no-session-persistence
   ```
   Переменная `CLAUDECODE` удаляется из env, чтобы разрешить вложенный вызов.
4. Claude Code читает PDF-файлы через инструмент `Read`, возвращает JSON в поле `result`
5. Временные файлы удаляются в блоке `finally`
6. Ответ распаковывается: если обёрнут в ```json...``` — снимается обёртка, затем `json.loads()`

### 4b. Анализ через Anthropic REST API (`analyze_with_claude`)

1. PDF кодируются в **base64**
2. Формируется `content[]` — список блоков: текстовые метки + PDF-документы (тип `document`, `media_type: application/pdf`) + итоговый текст с размером партии и базой оборудования
3. `client.messages.create(model="claude-sonnet-4-20250514", max_tokens=4096, system=SYSTEM_PROMPT, ...)`
4. Ответ: `message.content[0].text` → снятие обёртки ```json``` → `json.loads()`

### 5. Системный промпт (SYSTEM_PROMPT)

Определён как константа в `app.py`. Задаёт Клоду роль нормировщика и диктует:
- Что анализировать: чертёж (геометрия, материал, допуски), маршрутная карта (операции, оборудование)
- Что считать: `t_шт` (мин/дет) и `t_пз` (мин/партию — **полное**, не на деталь)
- Учитывать: материал, тип обработки, сложность геометрии, квалитет, тип оборудования
- Формат `режимы` по типу операции: металлорежущие `V/S/t/n`, сварочные `I/U/Vсв/d`, прочие `—`
- Использовать оборудование **только из переданной базы завода**
- **Приоритет цеха**: сначала искать оборудование в рабочем цеху детали (из МК), и только если нет нужного — в других цехах. В поле "оборудование" всегда указывать цех: `Станок [Цех №1]`
- Если станок взят из другого цеха — объяснить в обосновании почему
- Возвращать **строго JSON-массив**, без пояснений вне JSON

### 6. Структура ответа от Claude

Массив объектов, каждый содержит:
```json
{
  "деталь": "название из чертежа",
  "операция": "010 Токарная",
  "оборудование": "Токарный станок 16К20 [Цех №1]",
  "t_шт_предложено": 12.5,
  "t_пз_предложено": 15.0,
  "режимы": "V=120 м/мин, S=0.3 мм/об, t=1.5 мм, n=850 об/мин",
  "обоснование": "текст обоснования"
}
```

В консоль пишется `[DEBUG]` строка с операцией и полем `режимы` для каждой операции.

`/api/analyze` возвращает `{"operations": [...], "stub": false}`.

### 7. Отображение результатов (фронтенд)

- Результаты рендерятся в таблицу с редактируемыми полями `t_шт` и `t_пз`
- Фронтенд **пересчитывает t_пз на деталь**: `t_пз_на_деталь = t_пз_предложено / batch_size` — отображается рядом с полным t_пз
- Технолог может отредактировать значения в таблице

### 8. Подтверждение и сохранение (`/api/confirm`)

Технолог нажимает «Сохранить» → `fetch POST /api/confirm` с телом:
```json
{
  "operations": [...],
  "изделие": "необязательное название изделия"
}
```

Для каждой операции выполняется `INSERT INTO нормы` со всеми полями. `t_шт_подтверждено` и `t_пз_подтверждено` берутся из `t_шт_предложено` / `t_пз_предложено`, если технолог не менял значения (фронтенд передаёт оба поля). Поле `подтверждено = 1`. Возвращает `{"saved": N, "ids": [...]}`.

### 9. История (`/api/history`)

`fetch GET /api/history` → `SELECT * FROM нормы ORDER BY дата DESC LIMIT 200` → JSON-массив записей. Отображается в нижней части страницы.

## Важные детали

- Путь к Claude Code CLI захардкожен в `CLAUDE_BIN` (актуальный: `2.1.78/claude.app/Contents/MacOS/claude`) — при обновлении Claude Code менять вручную
- В корне есть два файла БД (`norming.db` и `нормирование.db`) — активный определяется константой `DB_PATH = "norming.db"`
- `USE_STUB=true` возвращает тестовые данные (STUB_OPERATIONS в app.py) без вызова API — удобно для разработки фронтенда
- `t_пз` Claude возвращает **на всю партию**; `batch_size` зафиксирован как 1, деление не производится
- `equipment_db.json` — 366 единиц оборудования (302 рабочих), кешируется в памяти на всё время жизни процесса — перезагрузка требует перезапуска сервера
- Все покрасочные камеры находятся в **Цех №3** (исправлено в equipment_db.json)
- Виртуальное окружение: `run.sh` использует `/tmp/norm_venv/`, папка `venv/` в корне не работает (пробел в пути)
